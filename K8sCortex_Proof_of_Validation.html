1. Overview & Objective

K8sCortex MCP Server exposes 75 Kubernetes management tools over the Model Context Protocol (MCP). The purpose of this document is to serve as an official proof of validation that all 75 tools work correctly against real clusters across multiple cloud providers — not just a local emulator — before publishing the server to the MCP marketplace.

The validation covered:

  • Workload management (Pods, Deployments, StatefulSets, DaemonSets, Jobs, CronJobs)
  • Configuration management (ConfigMaps, Secrets, ResourceQuotas)
  • Networking (Services, Ingress, NetworkPolicies)
  • Autoscaling (HPA, VPA)
  • Node operations (cordon, uncordon, drain, taint, pressure)
  • GitOps (ArgoCD app status, diff, sync)
  • Multi-cluster operations
  • Cloud-awareness tools (node pools, workload identity, PDB, storage classes, addon health, limit ranges)
  • Resource intelligence (metrics, recommendations)
  • Pod diagnostics (describe_pod with per-container crash state and events)
  • Rollout management (rollout_status, rollout_undo)
  • Network discovery (list_ingresses, get_service_endpoints)

1.0 How To Use This MCP

K8sCortex MCP is a governed Kubernetes operations interface for humans, automation, and AI runtimes. It exposes Kubernetes actions as MCP tools so users can run diagnostics, operations, and GitOps checks through a consistent tool contract instead of ad-hoc shell commands.

1.0.1 Primary Uses

  • Cluster diagnostics: list workloads, inspect logs, events, quotas, and node health
  • Day-2 operations: safe scale/restart/rollout and dry-run node operations
  • Governed changes: role-based access and structured audit logging for every tool call
  • GitOps visibility: Argo application status, drift, sync trigger, and comparison
  • Multi-environment parity: same tool contract across Minikube, GKE, AKS, and EKS

1.0.2 Quick Start (local)

cd /path/to/k8snexus-mcp
npm install
npm run build
DISABLE_AUTH=true PORT=3006 npm run dev

1.0.3 Basic MCP validation

# 70-tool core matrix (includes 7 Flux tools)
MCP_URL=http://localhost:3006/mcp npx tsx tests/e2e-all-63-minikube.ts

# 5-tool smoke matrix
MCP_URL=http://localhost:3006/mcp npx tsx tests/e2e-smoke-5-new-tools.ts

7. E2E Test Execution via MCP

7.1 How the test works

The validation uses two independent TypeScript MCP clients: tests/e2e-all-63-minikube.ts (main matrix, now extended to 70 tools including Flux) and tests/e2e-smoke-5-new-tools.ts (new-tool smoke test). Together they validate all 75 tools. Each script:

  1. Connects to the MCP server over HTTP (http://localhost:3007/mcp)
  2. Calls tools/list to discover all registered tools
  3. Calls each tool by name with real parameters via tools/call (standard MCP JSON-RPC)
  4. Asserts each response is non-error and contains expected content
  5. Prints PASS or FAIL per tool with reason
Important: The test script never calls kubectl or the Kubernetes API directly. Every single Kubernetes operation goes through the MCP server's HTTP transport — exactly as a real AI runtime client would call it.

7.2 Run commands

# Main 63-tool E2E matrix
MCP_URL=http://localhost:3007/mcp npx tsx tests/e2e-all-63-minikube.ts 2>&1

# Smoke test for 5 new tools (describe_pod, rollout_status, rollout_undo, list_ingresses, get_service_endpoints)
MCP_URL=http://localhost:3007/mcp npx tsx tests/e2e-smoke-5-new-tools.ts 2>&1

7.3 Call flow per tool

Test script
  → HTTP POST /mcp  (MCP JSON-RPC tools/call)
    → MCP Server (dist/index.js)
      → @kubernetes/client-node
        → GKE Kubernetes API Server (us-central1-a)
          → GKE workloads / GKE control plane
        ← JSON response
      ← Structured result
    ← MCP JSON-RPC response
  ← PASS / FAIL assertion
